第一部分：WHY - 我们为什么必须改变？

Slide 1: 混乱的现状 —— “烟囱式”的协议迷宫

传输壁垒： 蓝牙(BLE)有一套报文，USB HID有一套，Wi-Fi (TCP/WebSocket) 又是一套。换个传输通道，代码几乎重写。

状态机灾难： “发固件的同时来了个调亮度的请求怎么办？” 缺乏统一的流控和并发隔离，极易导致设备死机或串台。

黑盒调试： 出了 Bug 互相扯皮。前端说发了，固件说没收到。纯二进制流没有可观测性，排障如同大海捞针。

🗣️ 演讲者备注：
大家好，今天给大家讲讲我们为什么要推翻过去，搞这套全新的 AXTP。大家回想一下，每次我们研发一款新硬件，或者加上一种新的连接方式（比如从 BLE 换到 Wi-Fi），大家是不是都要扒一层皮？协议总是变来变去，调试极其痛苦。这就是我们目前的现状——烟囱式的灾难。

Slide 2: 核心痛点 —— 软硬件业务语言的严重“割裂”

文档与代码永远不同步： Word/Markdown 里写的协议，跟 C++ 甚至前端 TypeScript 代码里写的经常对不上。

两端思维的鸿沟：

固件工程师： 算计着每一字节的 RAM，讨厌解析复杂的长字符串和多重嵌套的 JSON。

前端/云端工程师： 习惯了结构化的 JSON、清晰的 API 字典，看到“密密麻麻的 Hex 二进制”就头疼。

没有统一的契约： 业务层想加个新功能，软硬件得拉会讨论半天“这个字节占几位”，毫无工程效能可言。

🗣️ 演讲者备注：
比技术更可怕的是沟通成本。固件团队和前端团队简直在说两门外语！没有一套“共同的业务语言”，导致我们开发一个简单的联动功能，沟通和联调成本占了 80%。我们需要一个能让所有端都舒服的“桥梁”。

第二部分：WHAT - AXTP 带来的架构革命

Slide 3: AXTP 不是“又一个旧协议”，而是一个 Protocol OS

AXTP (Adaptive eXtensible Transport Protocol) 是一个注册表驱动（Registry-Driven）的系统级基础设施。

一套语言，多端运行：

写一次业务逻辑，无缝跑在 TCP、WebSocket、USB HID、BLE、甚至无状态的 UART 串口上。

物理倒置，逻辑不变：

无论谁发起的物理连接（手机连设备，还是设备连云端 Agent），设备永远是逻辑 Server，控制端永远是逻辑 Client。

🗣️ 演讲者备注：
AXTP 不是我们随便写的一个格式表，它是一个“协议操作系统”。它的核心理念是：无论底层的物理公路是泥巴路（串口）还是高速公路（TCP），上面跑的货车（业务逻辑）必须是同一辆！

Slide 4: AXTP 的三大杀手锏

控制面与数据面彻底隔离（控制跑腿，数据裸奔）

RPC（控制面）： 负责鉴权、查能力、设置参数，高可靠。

STREAM（数据面）： O(1) 查表透传音视频和 OTA 固件，完全不消耗 CPU，榨干硬件性能！

0-RTT 极速握手与双向协商

OPEN 申请通道 -> Server ACCEPT 裁决 -> 连发 Hello 制定规则。

Client 在鉴权时主动上报自身容量限制 (Capacity)，设备端下发能力掩码 (Capability)。毫秒级完成双向互探。

零信任安全与细粒度控制 (ACL)

所有高危操作在完成 Identify 鉴权前绝对不暴露，保证设备无法被恶意脚本攻击。

🗣️ 演讲者备注：
我们把最脏最累的活儿给剥离了。业务逻辑绝对不允许污染底层数据流。此外，我们实现了最快的建连方式和最严格的安全守卫。不见兔子不撒鹰，没鉴权绝不暴露设备的内部能力。

第三部分：HOW - 我们如何实施与落地？

Slide 5: 降维打击 —— 域级特征掩码 (Domain-Scoped Mask)

抛弃臃肿的 JSON 键值对协商！

极致压缩： 将设备支持的几百个能力，按业务域 (Domain) 划分。通过极简的二进制位图 (Bitmask) 交互。

多端同构： * 固件端：消耗十几字节内存，纯位运算 (mask & bit)。

云端/前端：使用 Base64 字符串透明传输，保留极高可读性。

🗣️ 演讲者备注：
怎么让单片机和云端都爽？我们设计了“域级掩码”。在底层，它是极致压缩的几个字节，在前端，它是一串 Base64 字符串。大家再也不用去解析上百个 boolean 字段的庞大 JSON 了，一行位运算代码直接搞定所有的能力校验和事件订阅。

Slide 6: 唯一真理 —— Registry-Driven 与代码生成流

我们彻底改变开发范式！禁止手写协议解析代码！

输入源 (Input)： 在 registry/ 目录下用 YAML 定义所有错误码、方法、事件和能力。

自动生成 (Generator)： 一键运行工具，自动分发成果物！

生成 C++ 的枚举与权限拦截器 -> 给固件团队。

生成 TypeScript 类型 -> 给前端/云端团队。

生成 Wireshark Lua 插件 -> 零成本排错，看穿二进制黑盒！

生成 API 文档 (protocol.md)。

🗣️ 演讲者备注：
以后，我们的仓库架构必须是隔离的。协议怎么定？写 YAML。代码怎么来？机器自动生成！任何人都不能去手动修改 generated 目录下的代码，从源头消灭“文档与代码不一致”的问题。同时，我们连抓包工具都自动生成了，彻底消灭联调扯皮！

Slide 7: 拥抱未来 —— 天生对大模型友好 (AI-Native)

在 YAML 注册表中引入一等公民：AI Hints (AI 元数据)。

Generator 可以将协议直接导出为标准 MCP (Model Context Protocol) 工具描述 (Tool Schemas)。

我们的设备出厂即支持 AI Agent 零代码、无缝接入调度！

🗣️ 演讲者备注：
这是一个超前的设计。AXTP 不仅是给人读的，更是给 AI 读的。我们加入了语义标签，未来，大模型智能体可以直接“读懂”我们设备的协议，自动帮你写出控制代码，甚至自动编排智能场景。

Slide 8: 规范落地与治理（行动呼吁 Call to Action）

为了保护这个强大的 Protocol OS 不走向失控，我们必须建立秩序：

建立架构审查委员会 (ARB - Architecture Review Board)：

所有新业务域 (Domain) 的开启、核心 YAML 文件的变更，必须由委员会审批 (PR Review) 后才可合入主干。

遵循单向数据流原则：

业务线需求 -> 提审 YAML -> 触发生成器 -> 拿到 SDK 成果物进行业务开发。

立刻行动 (Next Steps)：

架构组拉通生成器 (Generator) 团队进行第一战役点亮。

C++ 团队与前端团队基于 MVP 注册表跑通 Demo 链路。
